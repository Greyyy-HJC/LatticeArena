# Wilson Loop Datasets

Gauge configurations are not stored in the repository (too large). Download or generate them as described below.

## Available Ensembles

### `test_small` - for development and testing
- Lattice: 8^3 x 16
- beta: 5.8
- Action: Wilson gauge
- Configs: 100
- Purpose: fast validation, unit tests

### `benchmark` - for official scoring
- Lattice: 20^3 x 40
- beta: 6.0
- Action: Wilson gauge
- Configs: 4000
- Purpose: benchmark scoring and leaderboard

## Generating Configs

Use the provided PyQUDA HMC script. The default command generates the `test_small`
ensemble under `tasks/wilson_loop/dataset/test_small/`:

```bash
.venv/bin/python tasks/wilson_loop/scripts/generate_configs.py \
    --latt 8,8,8,16 \
    --beta 5.8 \
    --n-configs 100 \
    --warmup 500 \
    --save-every 5 \
    --traj-length 1.0 \
    --n-steps 100 \
    --seed 10086 \
    --output tasks/wilson_loop/dataset/test_small \
    --resource-path .cache/quda
```

This generation path requires a working QUDA + PyQUDA installation. In practice
that also means `gmpy2` must be present because `pyquda-utils` imports it in the
HMC helper stack, and a matching CuPy wheel must be installed for the active
CUDA runtime. On CUDA 12 systems, install `cupy-cuda12x`. If the runtime is
incomplete, the script exits with setup instructions instead of failing during
normal repository import.

### Benchmark parameter contract

The official `benchmark` ensemble is documented for the same script, but is not
generated by default in this repository revision:

```bash
.venv/bin/python tasks/wilson_loop/scripts/generate_configs.py \
    --latt 20,20,20,40 \
    --beta 6.0 \
    --n-configs 4000 \
    --warmup 4000 \
    --save-every 5 \
    --traj-length 1.0 \
    --n-steps 100 \
    --seed 10086 \
    --output tasks/wilson_loop/dataset/benchmark \
    --resource-path .cache/quda
```

## File Format

Configs are stored as numpy `.npy` files with shape
`(Nd, Lx, Ly, Lz, Lt, Nc, Nc)` and dtype `complex128`, where Nd=4 (x,y,z,t
directions) and Nc=3 (SU(3) colors).

This on-disk ordering is the Wilson-loop task contract and intentionally differs
from PyQUDA's native NPY gauge ordering `(Nd, Lt, Lz, Ly, Lx, Nc, Nc)`.

Generated directories also include:

- `metadata.json` with lattice/action/HMC settings and package versions
- `manifest.csv` with one row per saved configuration, including trajectory,
  plaquette, and maximum link deviation from the unit gauge

After generation, check that the run actually evolved away from the cold start:

- `metadata.json` contains `evolution_detected`
- `metadata.json` contains `post_warmup_acceptance_rate`,
  `saved_configs_accepted`, and `unique_saved_plaquette_count`
- `manifest.csv` contains `max_link_deviation_from_identity`

If these remain zero and the plaquette stays at `1.0`, the saved configs should
not yet be treated as a real pure-gauge ensemble. In practice this usually
means your run never moved away from the cold start, so increase warmup or
inspect the local PyQUDA/QUDA HMC runtime.

For quick development subsets, also watch the post-warmup acceptance diagnostics.
If `saved_configs_accepted` is near zero or `unique_saved_plaquette_count` is
much smaller than `saved_configs_total`, you are mostly saving repeated gauges
after rejected trajectories. On the local RTX 3060 setup used for smoke tests,
smaller MD step sizes such as `traj_length / n_steps <= 0.01` behaved much more
reliably than coarse quick-test settings.
